<LessonMeta level="Advanced" duration="1 week" track="Module 13" prerequisites="Modules 01-12" />

This final module turns technical work into professional presentation. Students learn how backend work is reviewed, documented, and discussed in team environments — the soft-wrapped layer that makes the rest of the curriculum hireable.

<Objectives>
- Open a pull request with a clean commit history and a review-ready description
- Give and receive code review feedback that improves the code without bruising the author
- Package a capstone repository so a stranger can run it and understand it
- Answer backend interview questions about event loop, auth, architecture, and trade-offs with confidence
</Objectives>

## What this module covers

- Git fundamentals including add, commit, branch, merge, and rebase
- pull requests, code reviews, and collaboration workflows
- conventional commits and release note awareness
- backend project README writing
- portfolio-ready capstone packaging
- interview preparation for Node.js backend roles

## Teaching sequence

1. Review branch and commit discipline.
2. Run a pull request workflow on real code.
3. Improve project documentation and setup clarity.
4. Practice presenting a capstone and defending implementation choices.

## Live examples

- a clean pull request with reviewable commit history
- README structure for backend onboarding
- architecture walkthrough for a capstone project

## Labs

- prepare a capstone repository for public review
- perform peer code review on another student project
- write interview answers for event loop, auth, and architecture questions

## Exit outcomes

- students can collaborate like junior backend engineers
- students finish with projects that are understandable to reviewers
- students can speak clearly about what they built and why

<Callout type="tip" title="The README is the interview">
Reviewers and interviewers rarely read the code before the README. If the README does not clearly state what the project is, how to run it, and what decisions were made, the code may as well not exist. Polish it like it will be the first thing a hiring manager sees — because it will be.
</Callout>
